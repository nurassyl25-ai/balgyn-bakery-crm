# Balgyn Bakery CRM

Полноценный прототип CRM для кондитерского магазина: заказы, доставки/самовывоз,
клиенты с напоминаниями о днях рождения. Состоит из двух частей:

- **backend/** — Python (FastAPI) + SQLite. Вся бизнес-логика (расчёт статусов
  заказов, дни рождения, "зависшие" заказы) лежит в одном файле `backend/logic.py` —
  именно его редактируют, чтобы поменять правила работы CRM.
- **src/** — React-фронтенд (тот же интерфейс, что вы уже видели), обращается
  к бэкенду по HTTP.

## Запуск локально

### 1. Бэкенд (Python)

```bash
cd backend
python -m venv venv
source venv/bin/activate        # Windows: venv\Scripts\activate
pip install -r requirements.txt
uvicorn main:app --reload --port 8000
```

Сервер поднимется на http://localhost:8000. При первом запуске автоматически
создаётся файл базы данных `balgyn.db` и заполняется демо-данными.

Проверить, что сервер работает: откройте http://localhost:8000/api/health —
должно вернуть `{"status": "ok"}`. Также доступна интерактивная документация
API: http://localhost:8000/docs

### 2. Фронтенд (в отдельном терминале)

```bash
npm install
npm run dev
```

Откроется на http://localhost:5173 и будет обращаться к бэкенду на порту 8000.

## Авторизация

При первом запуске бэкенд создаёт две демо-учётные записи:

| Логин   | Пароль      | Роль      |
|---------|-------------|-----------|
| manager | balgyn2026  | Менеджер  |
| baker   | balgyn2026  | Кондитер  |

**Перед реальным использованием обязательно:**
1. Смените пароли (через `/api/auth/change-password` или напрямую в базе)
2. Задайте свой `JWT_SECRET_KEY` через переменную окружения — иначе используется
   значение по умолчанию, которое не подходит для продакшена:
   ```bash
   export JWT_SECRET_KEY="сгенерируйте-длинную-случайную-строку"
   ```

Роль пользователя (Менеджер / Кондитер) теперь определяется учётной записью,
а не переключателем в интерфейсе — кондитер физически не может зайти в раздел
«Клиенты» или отредактировать чужие данные, это проверяется на сервере.



Откройте `backend/logic.py`. Там четыре вещи, которые можно поменять:

- `STALE_DAYS` — через сколько дней заказ без движения считается "зависшим"
- `SOON_THRESHOLD_DAYS` — через сколько дней до срока заказ помечается "скоро"
- `BIRTHDAY_REMINDER_DAYS` — за сколько дней показывать напоминание о ДР клиента
- `compute_order_status()` и `days_to_next_birthday()` — сами функции расчёта,
  если нужно что-то более сложное (например, разные пороги для разных типов
  изделий)

После изменения файла просто сохраните — если бэкенд запущен с `--reload`,
изменения подхватятся автоматически, без перезапуска.

## Сборка фронтенда для продакшена

```bash
npm run build
```

Готовые файлы появятся в папке `dist/`.

## Деплой

### Бэкенд (Python)
Подходят Render.com или Railway.app — оба поддерживают Python из коробки и
имеют бесплатный тариф. Команда запуска: `uvicorn main:app --host 0.0.0.0 --port $PORT`

Важно: SQLite-файл на бесплатных тарифах может не сохраняться между
перезапусками контейнера. Для реальной эксплуатации стоит подключить
PostgreSQL (Render/Railway дают его бесплатно) — в `backend/database.py`
нужно будет только поменять строку подключения.

### Фронтенд
Vercel или Netlify (как и раньше). В настройках проекта укажите переменную
окружения `VITE_API_URL` со ссылкой на задеплоенный бэкенд.

## Что это даёт по сравнению с предыдущей версией

- Данные теперь в одной базе на сервере — доступны с любого устройства и
  любому сотруднику одновременно (не привязаны к браузеру одного человека)
- Бизнес-правила — в понятном Python-файле, редактируются без прикосновения
  к React-коду
- Есть готовая точка роста: авторизация пользователей, роли с паролями,
  экспорт отчётов, автоматические напоминания о днях рождения по WhatsApp/email
